# Synthetic ledger — protocol-violation test fixture

`violations-case-ledger.jsonl` is a **hand-authored** case ledger (NOT generated by
the container demo) whose sole purpose is to exercise the Log Replay demo's
protocol-violation flagging (the red outline + ⚠️ rendered by
`ui/src/App-logreplay.tsx`, driven by the `violation` flag the validating mapper
sets in `ui/src/utils/caseLedgerMapper.ts`).

These committed fixtures live under `ui/src/sample-logs/` (they ship with the app
and are imported by the "Load …" buttons), NOT under the gitignored repo-root
`devlogs/` (which is runtime output from the container demo).

The committed `two-actor` sample is a clean happy path, so it never
flags anything. This fixture deliberately includes several illegal transitions
across multiple state machines (RM, EM-terminate, and a case-level PXA jump),
chosen so each illegal step lands in a handler that emits a node (a pure
status-only violation would be logged but produce no visible node — see the
mapper's `changes.length === 0` early return).

The PXA entry (logIndex 7) specifically covers the **case-level EM/PXA validation
path** (`applyCaseLevelForward`), which for a long time silently ignored illegal
case-level jumps as if they were merely stale snapshots. The distinction the
mapper now makes: a snapshot reachable *backward* from the current state is stale
(ignored); a snapshot reachable in *neither* direction is a genuine violation
(flagged). PXA is monotonic, so `pxA` (attacks, not public) and `Pxa` (public, no
attacks) are mutually unreachable — a perfect "neither-direction" test case. (EM
has no such incomparable pair — it funnels toward the EXITED sink — so EM illegal
jumps surface via the `remove_embargo` handler instead, as at logIndex 5.)

## How to load it

In the Log Replay demo, click the red **⚠️ Load Violation Sample** button — it
imports `violations-case-ledger.jsonl` directly. (The green "Load Sample Case"
button loads the happy-path two-actor sample; "Load FVV Case" loads the 3-party
sample.) You can also **Select Log Files** and pick this file manually from
`ui/src/sample-logs/synthetic/`.

## Companion fixture: `inferred-multistep-case-ledger.jsonl`

A second hand-authored fixture exercises the **inferred-step tripwire** (the
amber dashed outline + ℹ️, driven by the `inferred` flag — distinct from the red
`violation` flag). Because the log records snapshots, not transitions, a status
diff that spans more than one legal step means intermediate states were not
logged and the mapper *inferred* the legal path between them. That inference
could in principle mask an illegal 1-step jump that happens to be bridgeable by a
longer legal detour, so the mapper marks such nodes "inferred, not observed."

This fixture: after the offer seeds vendor-1 at `RM=RECEIVED, VFD=Vfd`, a single
status snapshot jumps straight to `ACCEPTED VFD` — a double multi-hop
(RM `RECEIVED →validate→ VALID →accept→ ACCEPTED`; VFD `Vfd →fix_is_ready→ VFd
→fix_is_deployed→ VFD`). Both are legal paths, so **no violation** — the node
renders with the amber ℹ️ annotation and the tooltip explains each inferred path.

> Real generator output does NOT currently produce multi-hop diffs (validate has
> its own `validate_report` verb; only ACCEPTED/CLOSED ever appear as RM status
> snapshots; VFD milestones are logged one at a time). So this tripwire is dormant
> against today's logs — it exists to surface the inference *if* the generator's
> snapshot granularity ever coarsens. This fixture is the only thing that
> currently triggers it.

## Entries and expected result

| logIndex | eventType | derived trigger | shadow source | legal? |
|---|---|---|---|---|
| 0 | offer_case_manager_role | seeds (vendor RM=RECEIVED/VFD=Vfd, finder RM=ACCEPTED, EM=ACTIVE) | — | seed |
| 1 | validate_report | rm `validate` | RECEIVED → VALID | ✅ legal |
| 2 | validate_report | rm `validate` | VALID | ❌ **VIOLATION** (no `validate` from VALID) |
| 3 | add_participant_status | rm `accept` + vfd `fix_is_ready` | VALID→ACCEPTED, Vfd→VFd | ✅ legal (multi-advance) |
| 4 | remove_embargo_event_from_case | em `terminate` | ACTIVE → EXITED | ✅ legal |
| 5 | remove_embargo_event_from_case | em `terminate` | EXITED | ❌ **VIOLATION** (no `terminate` from EXITED) |
| 6 | add_participant_status | pxa `attacks_are_observed` | pxa → pxA | ✅ legal (case-level PXA advance) |
| 7 | add_participant_status | pxa (illegal) | pxA → Pxa | ❌ **VIOLATION** (pxA and Pxa mutually unreachable) |

**Expected:** 8 nodes on the timeline; the nodes for **logIndex 2** (RM),
**logIndex 5** (EM-terminate), and **logIndex 7** (case-level PXA) render with the
red violation outline + ⚠️, and the event log shows a `PROTOCOL VIOLATION` line for
each. The legal entries (0, 1, 3, 4, 6) render normally — so the fixture also
confirms flagging is *targeted*, not blanket, and that it covers the case-level
EM/PXA validation path in addition to the per-participant RM/VFD path.
