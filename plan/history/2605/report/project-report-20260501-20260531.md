# Vultron Project — May 2026 Work Sponsor Report

## Executive Summary

May 2026 was the project's highest-throughput month to date, with 30 pull
requests merged spanning protocol automation, multi-actor demonstration
scenarios, architecture hardening, and extensive bug fixing. The flagship
achievement was bringing the two-actor coordinated vulnerability disclosure
(CVD) demo to full lifecycle completion — two independent actors can now
execute the entire disclosure workflow from report submission through
publication. Significant investment also went into migrating procedural
protocol logic into auditable Behavior Tree (BT) structures, and into
hardening the architecture to enforce clean layer boundaries.

---

## Multi-Actor Demo: Full Lifecycle Achievement

The project reached a major milestone: a two-actor demo that executes the
complete CVD lifecycle (report → case creation → embargo negotiation →
fix development → public disclosure), with each actor running as an
independent service communicating over HTTP.

Key work included implementing the core two-actor demo capabilities
([PR #465](https://github.com/CERTCC/Vultron/pull/465)), completing the
full vulnerability-fix-deploy-publish-announce (VFDPxa) lifecycle
([PR #463](https://github.com/CERTCC/Vultron/pull/463)), and automating
Case Actor spawning with case-manager role delegation
([PR #473](https://github.com/CERTCC/Vultron/pull/473),
[PR #468](https://github.com/CERTCC/Vultron/pull/468)). Shared demo
helpers were extracted into a reusable library
([PR #541](https://github.com/CERTCC/Vultron/pull/541)), and a GitHub
Actions CI workflow now builds and runs the demo containers on every PR
([PR #591](https://github.com/CERTCC/Vultron/pull/591)). Bootstrap
sequence integration tests verify the end-to-end case creation and
late-joiner flows
([PR #544](https://github.com/CERTCC/Vultron/pull/544),
[PR #545](https://github.com/CERTCC/Vultron/pull/545)).

## Protocol Automation via Behavior Trees

Several protocol-significant workflows were migrated from procedural code
into Behavior Tree (BT) structures — composable, auditable decision trees
that make protocol logic inspectable independently of implementation
details.

The participant-status handler was refactored into a full BT
([PR #525](https://github.com/CERTCC/Vultron/pull/525)), sync operations
gained their own behavior trees
([PR #456](https://github.com/CERTCC/Vultron/pull/456)), and the
embargo-announcement handler was implemented as a BT
([PR #590](https://github.com/CERTCC/Vultron/pull/590)). All sender-side
trigger use cases were refactored to use a shared `SenderSideBT` pattern
([PR #619](https://github.com/CERTCC/Vultron/pull/619)), and case-log
cascading now flows through BT nodes rather than post-execution
procedural calls
([PR #620](https://github.com/CERTCC/Vultron/pull/620)). A comprehensive
BT idiom audit identified 10 remaining areas where procedural patterns
still bypass BT structure, tracked as a concern for future work.

## Architecture Hardening

Clean separation between protocol core, wire format, and adapter layers
was enforced across multiple fronts. Architecture violations where core
modules imported from wire or adapter layers were systematically removed
([PR #444](https://github.com/CERTCC/Vultron/pull/444)). Deprecated
compatibility methods were cleaned up, and data-layer operations were
migrated to typed interfaces. Participant triggers now correctly address
only the Case Actor rather than broadcasting directly, enforcing the
canonical communication model
([PR #616](https://github.com/CERTCC/Vultron/pull/616)).

Cyclomatic complexity was reduced project-wide and a gate enforced at a
maximum of 10 per function, ensuring no new high-complexity code is
introduced. Data-layer isolation was fixed so each actor uses its own
storage instance
([PR #536](https://github.com/CERTCC/Vultron/pull/536)), and per-app
singleton sharing was eliminated
([PR #540](https://github.com/CERTCC/Vultron/pull/540)). Four priority
groups — Architecture Hardening, Trigger Classification, Cyclomatic
Complexity Enforcement, and RAFT Consensus — were completed and archived
with no remaining open issues.

## Bug Fixes and Reliability

Over a dozen bugs were fixed, many discovered during multi-actor demo
testing. Actor IDs were corrected to use proper hostnames instead of
localhost
([PR #555](https://github.com/CERTCC/Vultron/pull/555)), and a
double-prefix bug in HTTP path construction was resolved
([PR #559](https://github.com/CERTCC/Vultron/pull/559)). Case
participants were not being stored in replicas
([PR #564](https://github.com/CERTCC/Vultron/pull/564)) and embedded
participants in Announce activities were being silently dropped
([PR #567](https://github.com/CERTCC/Vultron/pull/567),
[PR #577](https://github.com/CERTCC/Vultron/pull/577)). The reporter
participant now correctly starts at the expected state
([PR #621](https://github.com/CERTCC/Vultron/pull/621)), and embargo
consent reset now tolerates inline participant objects
([PR #625](https://github.com/CERTCC/Vultron/pull/625)). A pytest
timeout guardrail (30 seconds per test) was added to catch runaway tests
early ([PR #529](https://github.com/CERTCC/Vultron/pull/529)), and test
suite performance was improved by eliminating unnecessary retry backoff
([PR #623](https://github.com/CERTCC/Vultron/pull/623)).

## Developer Tooling and Agent Infrastructure

The project's AI-assisted development workflow matured significantly.
`AGENTS.md` was restructured into per-directory files to reduce root file
size and improve agent context targeting
([PR #581](https://github.com/CERTCC/Vultron/pull/581)). A new
`ingest-concern` skill automates converting tracked concerns into
implementation plans
([PR #587](https://github.com/CERTCC/Vultron/pull/587)). A stale-claim
sweeper workflow automatically detects and clears abandoned agent work
claims on GitHub Issues
([PR #429](https://github.com/CERTCC/Vultron/pull/429)), and issue
hierarchy enforcement was added to maintain parent–child relationships
([PR #448](https://github.com/CERTCC/Vultron/pull/448)).

## Specification and Design Documentation

Multiple design ideas were formally ingested into specifications and
design notes, covering case creation sequencing, demo CI workflows,
documentation build optimization, and the distinction between
specifications and architectural decision records. Three concern issues
were investigated and resolved into concrete implementation plans
covering case-actor routing, AGENTS.md structure, and demo construction
gaps.

## Dependencies

Six automated dependency bumps were merged cleanly during the period,
including updates to `fastapi`, `uvicorn`, `black`, `idna`, and
`pymdown-extensions`.
