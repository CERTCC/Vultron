Fixed three slow test files that together accounted for >70 seconds of test
suite runtime (now <5 seconds combined).

**Root causes and fixes:**

1. `test/architecture/test_activity_factory_imports.py` (27s → 1s):
   `_collect_violations()` called `REPO_ROOT.rglob("*.py")` which scanned
   `.venv/` (6 726 Python files) in addition to `vultron/` and `test/`.
   Fix: restrict scan to `_SCAN_ROOTS = ("vultron", "test")`.

2. `test/metadata/test_append_history.py` (8.6s → 0.72s):
   Each of 38 tests spawned a Python subprocess (~0.2–0.4 s startup each).
   Fix: replaced `subprocess.run(sys.executable -m ...)` with an in-process
   helper that patches `sys.argv`, `sys.stdin`, and redirects stdout/stderr
   before calling `main()` directly.  Introduced `_RunResult` dataclass to
   preserve the existing test interface.

3. `test/adapters/driving/fastapi/routers/test_trigger_report.py` (39s → 2s):
   `outbox_handler` made real HTTP requests to non-existent URLs
   (`https://example.org/actors/vendor/inbox/`, `urn:uuid:.../inbox/`) and
   retried with exponential backoff (0.5 + 1.0 + 2.0 s per test).
   Fix: added a module-level `autouse=True` fixture that patches
   `outbox_handler` with a no-op `AsyncMock` for every test.
   `TestTriggerReportOutboxScheduling` tests are unaffected — their
   `_make_patches()` context manager overrides the autouse mock for the
   duration of each delivery-assertion test.
