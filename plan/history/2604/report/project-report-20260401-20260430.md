# Vultron Project — April 2026 Project Report

## Executive Summary

April 2026 was a highly productive month for the Vultron project. The team
completed significant advances across three areas: the protocol automation
layer now handles a much larger share of the coordinated vulnerability
disclosure (CVD) workflow automatically, without manual scripting steps; the
project's internal specification infrastructure was overhauled to enable
machine-readable validation and AI-assisted development; and the developer
tooling that supports ongoing work was substantially improved. Multi-actor
demo scenarios were also hardened in meaningful ways, though some issues
surfaced post-close; that work continues in May.

---

## More of the CVD Protocol Now Runs Automatically

The most user-visible change of the month is that Vultron now handles more
of the end-to-end CVD workflow without manual intervention.

Previously, when a vulnerability report was received and validated, a human
operator (or demo script) had to manually trigger several follow-on steps:
engaging the case, adding the finder as a participant, and initializing an
embargo. These are now performed **automatically** by the Behavior Tree
engine the moment a report is validated. The system creates the case, sets
up a default embargo (active immediately, not just proposed), registers both
the finder and the vendor as participants, and queues the appropriate
notifications — all as a single atomic sequence.

Similarly, when an actor accepts a case invitation, the system now
**automatically engages** that actor in the case, without requiring a
separate manual step.

These improvements reduce the gap between Vultron's prototype behavior and
what a real-world CVD coordination system would need to do.

## Progress on Multi-Actor Coordination Scenarios

Vultron supports demonstration scenarios with two, three, and multiple actors
(vendors, finders, coordinators). In April, meaningful progress was made on
the reliability and realism of these scenarios:

- **Scenario scripts now drive the system the same way a real participant
  would** — by calling trigger endpoints rather than injecting messages
  directly into inboxes. This makes the demonstrations more realistic and
  exposes any gaps in the protocol's trigger coverage.

- **Multi-party embargo acceptance was improved.** A bug was fixed where
  the second (and subsequent) participant accepting an embargo would
  receive a 409 error and trigger an invalid embargo state transition. Each
  participant now records consent independently without interfering with the
  shared embargo state.

- **Automated integration tests** were added for all three multi-actor
  scenarios (two-actor, three-actor, multi-vendor), runnable via a single
  command with Docker Compose.

Some issues with the multi-actor scenarios were identified after the month
closed; those are being addressed as part of May's work.

## The Protocol API Has a Cleaner Architecture

A new **TriggerService** interface was introduced as a clean boundary between
the protocol logic and the HTTP delivery layer. This means the core
coordination logic can now be tested and reasoned about independently of
FastAPI or any specific web framework. It also gives future developers a
single, well-defined entry point for initiating protocol actions, rather than
having to understand the internal wiring of multiple adapter layers.

See [PR #411](https://github.com/CERTCC/Vultron/pull/411) and
[PR #409](https://github.com/CERTCC/Vultron/pull/409).

---

## Specifications Are Now Machine-Readable and Validated

Vultron's formal protocol requirements — previously maintained as Markdown
documents — were migrated to a structured YAML format with a Pydantic schema,
an automated linting tool, and continuous integration checks. Specification
IDs are now validated for consistency across the codebase.

In practical terms, this means:

- Specifications can be queried, cross-referenced, and exported
  programmatically.
- Violations (missing specs, broken references, malformed IDs) are caught
  automatically before code is merged.
- AI coding assistants working on the project now receive a richer,
  more structured view of the requirements, improving the quality of
  generated code.

See [PR #393](https://github.com/CERTCC/Vultron/pull/393) and
[PR #399](https://github.com/CERTCC/Vultron/pull/399).

## Developer Workflow Is More Structured and Sustainable

The agent-assisted development workflow that has been supporting the project
was significantly improved. Project history is now recorded in structured,
per-entry files that can be searched and queried, rather than a single
ever-growing document. Skills for reviewing priorities, promoting lessons
learned into formal specs, and managing the implementation backlog were
added or improved. The root documentation file used to guide AI agents was
condensed from ~1,700 lines to ~400 lines, improving reliability and reducing
overhead.

See [PR #406](https://github.com/CERTCC/Vultron/pull/406) and
[PR #399](https://github.com/CERTCC/Vultron/pull/399).

## Bug Fixes and Quality Improvements

A focused bug-fix pass resolved seven issues that had been blocking or
degrading demo reliability:

- Fixed a serialization bug that caused case log entries to lose data when
  transmitted between actors.
- Fixed parsing of Accept/Reject responses so that nested invitation objects
  are correctly preserved.
- Fixed actor ID normalization in outbound message routing.
- Improved container log output to make the flow of a CVD coordination
  session legible when watching live logs.
- Fixed the default embargo to initialize as **active** rather than merely
  proposed — matching the intended protocol semantics.

See [PR #407](https://github.com/CERTCC/Vultron/pull/407).

---

## Documentation and Configuration

Versioning documentation was updated to describe the project's adopted
CalVer scheme. Several LaTeX rendering issues in the formal protocol
documentation were fixed. Seed configuration files were migrated from
JSON to YAML, laying groundwork for a more ergonomic configuration system.

See [PR #408](https://github.com/CERTCC/Vultron/pull/408),
[PR #388](https://github.com/CERTCC/Vultron/pull/388), and
[PR #389](https://github.com/CERTCC/Vultron/pull/389).
