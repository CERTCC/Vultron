Implemented the TriggerService facade to eliminate the redundant
`_trigger_adapter.py` shim layer and enable direct domain-layer testing
without an HTTP stack.

This history entry was recorded in April 2026; RFC-400 itself was authored
earlier.

## Files created

- `vultron/core/ports/trigger_service.py` — `TriggerServicePort` Protocol
  with 19 method signatures (inbound driving port)
- `vultron/core/use_cases/triggers/service.py` — `TriggerService` concrete
  implementation; each method delegates to the corresponding `SvcXxx`
  use-case class
- `vultron/adapters/driving/fastapi/deps.py` — centralized FastAPI DI:
  `get_trigger_dl`, `get_canonical_actor_dl`, `get_trigger_service`

## Files modified

- All 5 trigger routers migrated to `Depends(get_trigger_service)` +
  `domain_error_translation()`; outbox handler receives shared DL as
  third positional arg via `get_datalayer()`
- All 5 HTTP test files updated to override `get_trigger_service`
- `test_trigger_services.py` migrated to call `TriggerService(dl).method()`
  directly with `VultronError`/`PydanticValidationError` assertions

## Files deleted

- `vultron/adapters/driving/fastapi/_trigger_adapter.py`

## Outcome

All 1991 tests pass; Black, flake8, mypy, pyright clean. RFC-400.1,
RFC-400.2, RFC-400.3, RFC-400.5 complete. RFC-400.4 deferred (covered by
migrated test_trigger_services.py).
