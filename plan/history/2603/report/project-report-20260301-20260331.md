# Vultron Project — March 2026 Work Sponsor Report

## Executive Summary

March 2026 was the most productive month of the project to date. The team
completed a major structural overhaul of the codebase (the "hexagonal
architecture" refactor), shipped the first working multi-actor networked
demonstration, and significantly hardened both code quality and CI security.
Six priority groups were formally closed, over 53 pull requests were merged,
and the test suite grew from roughly 900 to 1,080 passing tests.

---

## Hexagonal Architecture Refactor Completed

The most significant effort of the month was completing the full
ports-and-adapters refactor of the Vultron codebase — a rearchitecting effort
that had been in progress since February. The goal was to ensure the protocol's
core domain logic (the rules governing how vulnerability disclosures are
coordinated) has no direct knowledge of external protocols, wire formats, or
databases. External concerns are now connected via well-defined interfaces
called "ports."

Concretely, this included:

- Moving the ActivityStreams message vocabulary (the wire format Vultron uses
  to communicate between actors) fully into a dedicated wire layer, cleanly
  separated from the domain core.
- Introducing a formal inbound-message pipeline: raw messages now flow through
  a parser, then a semantic extractor, then a dispatcher, then a use-case
  handler — each step isolated from the next.
- Relocating the behavior automation (Behavior Trees — the decision-logic trees
  that drive protocol automation) to the core layer and removing all
  wire-format imports from them.
- Completing a use-case interface standard: every protocol operation is now
  implemented as a uniform class (accepting a data-layer connection and a
  request event, and exposing a single `execute()` method).
- Removing all remaining architectural violations that had been tracked as
  technical debt.

Six priority groups were formally closed: PRIORITY-50 through PRIORITY-75.

## Multi-Actor Message Delivery and Actor Isolation

With the architecture clean, the team implemented the infrastructure needed
for multiple Vultron actors to exchange messages with each other across a
network:

- An **outbox delivery pipeline** (OX-1.0–1.4) was built: actors can now send
  activities outbound to other actors' inboxes via a dedicated delivery adapter.
  In-process actors (e.g., a CaseActor running in the same server) are routed
  via the in-process ASGI path; remote actors are routed via HTTP.
- **Per-actor DataLayer isolation** (ACT-1/2/3) was implemented: each actor
  running in a process now has its own independent data store, preventing
  state leakage between actors. An Architecture Decision Record (ADR-0012)
  documents this design choice.
- **CaseActor broadcast** (CA-1/2/3) was completed: when a case's state
  changes, the CaseActor automatically notifies all registered participants.
  A new action-rules endpoint allows actors to query what protocol actions are
  available to them in the current case state.

See [PR #337](https://github.com/CERTCC/Vultron/pull/337),
[PR #338](https://github.com/CERTCC/Vultron/pull/338), and
[PR #341](https://github.com/CERTCC/Vultron/pull/341).

## First Multi-Container Demo Scenario

At the end of the month, the team completed the first deterministic multi-actor
end-to-end demonstration of Vultron (D5-1 and D5-2):

- A multi-container Docker Compose configuration was built, allowing a
  Finder actor and a Vendor actor to run in separate containers and exchange
  messages over a real HTTP network.
- The demonstration exercises the full Finder→Vendor workflow: report
  submission, report validation, case engagement, case invite, and acceptance.
- The architecture review (D5-1) confirmed that all prerequisite architectural
  work is complete and the remaining gaps are operational (Docker setup, actor
  seeding) rather than protocol-level.

A core defect discovered during the demo (the Vendor participant record was
not created during report validation, which blocked later engagement) was
identified and fixed as part of this work.

See [PR #349](https://github.com/CERTCC/Vultron/pull/349) and the D5-1/D5-2
entries in the implementation history.

## Code Quality and Naming Convention Enforcement

Several quality-of-life improvements were made to reduce future maintenance
burden:

- **Naming convention enforcement (NAMING-1)**: A large rename touched 130
  Python files (~2,756 occurrences) to enforce a consistent convention for
  wire-layer field names. The change is invisible to external callers (JSON
  serialization/deserialization is unchanged) but eliminates a confusing
  inconsistency in the internal code.
- **Warnings treated as errors (QUALITY-1)**: The test suite was configured
  to fail on any Python warning, not just test failures. This surfaced and
  fixed a file-handle leak in the database adapter.
- **Use-case reorganization (REORG-1)**: Inbound message handlers and query
  operations were split into separate sub-packages for clarity.
- **Mypy and Pyright lint burn-down**: The static type-checking baseline was
  significantly reduced, improving type safety across the codebase.

See [PR #354](https://github.com/CERTCC/Vultron/pull/354),
[PR #350](https://github.com/CERTCC/Vultron/pull/350), and
[PR #342](https://github.com/CERTCC/Vultron/pull/342).

## CI Security Hardening

All GitHub Actions used in the CI/CD pipeline were migrated to reference
specific immutable commit SHAs (replacing version tags that can be moved or
replaced). This follows supply-chain security best practices: a pinned SHA
guarantees the exact code that runs in CI cannot be silently swapped.

Automated tests were added to the test suite (53 new tests) that verify every
`uses:` line in every workflow file is pinned to a full SHA and annotated with
a human-readable version comment. An Architecture Decision Record (ADR-0014)
documents the policy.

See [PR #339](https://github.com/CERTCC/Vultron/pull/339),
[PR #340](https://github.com/CERTCC/Vultron/pull/340), and
[PR #351](https://github.com/CERTCC/Vultron/pull/351).

## Specification Infrastructure Improvements

The machine-readable project specifications (YAML files that formal
requirements are tracked in) received a three-part audit:

- **SPEC-AUDIT-1**: Eliminated redundant requirements by adding
  cross-references between related requirements in different spec files,
  reducing maintenance overhead.
- **SPEC-AUDIT-2**: Normalized the strength of obligation language (MUST,
  SHOULD, MAY) across all spec files to follow RFC 2119 definitions
  consistently.
- **SPEC-AUDIT-3**: Fixed stale file paths and symbol references that had
  drifted since earlier refactoring.

See [PR #353](https://github.com/CERTCC/Vultron/pull/353),
[PR #355](https://github.com/CERTCC/Vultron/pull/355), and
