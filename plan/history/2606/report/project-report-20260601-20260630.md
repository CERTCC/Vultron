# June 2026 Project Report

## Executive Summary

June 2026 was the most active month to date, with over 130 PRs merged across all layers of the stack. Work was dominated by two intertwined structural drives: porting the demo fuzzer layer to py_trees probabilistic stubs, and hardening the behavior-tree/ledger execution contract with strict ordering invariants and commit-path uniqueness. A major refactor completed the BT node module reorganization (splitting flat `nodes.py` files into semantic subpackages across all remaining areas), while the Case Proposal protocol received its full trigger-side implementation. The month closed with new spec requirements and AGENTS.md guidance capturing hard-won patterns from production incidents.

---

## Demo Fuzzer Layer: py_trees Probabilistic Simulation Stubs (FUZZ-01–08a)

The full fuzzer port from legacy `vultron/bt/*/fuzzer.py` modules to `vultron/demo/fuzzer/` was completed across seven implementation issues ([#860](https://github.com/CERTCC/Vultron/pull/1105)–[#866](https://github.com/CERTCC/Vultron/pull/1116)), plus a catalog-pass and human-audit round (FUZZ-08a, FUZZ-08a-ter, FUZZ-08a-quat, [PR #1179](https://github.com/CERTCC/Vultron/pull/1179), [PR #1195](https://github.com/CERTCC/Vultron/pull/1195), [PR #1202](https://github.com/CERTCC/Vultron/pull/1202)).

Each node is now a `WeightedBehavior` subclass (`ProbablyFail`, `UsuallySucceed`, etc.) with docstrings covering semantic function, success probability, input category, and automation potential, satisfying BT-16-003 and BT-16-005. The ~100-node catalog spans embargo management, report validation/prioritization, VUL ID, fix development, publication, and report-to-others workflows. A final shape-correction pass reclassified 47 nodes (Sentinel→Evaluator, Sentinel→ProtocolInternal) after human audit. Demo wiring of these stubs into multi-actor scenarios is deferred to issue #867.

## Behavior Tree / Ledger Execution Invariants

A major cluster of work enforced strict execution contracts around behavior-tree commit ordering and ledger correctness:

- **DL mutation ratchet and AST invariant** ([PR #1088](https://github.com/CERTCC/Vultron/pull/1088), [PR #1078](https://github.com/CERTCC/Vultron/pull/1078)): `execute()`-phase DataLayer mutations are gated by an AST-layer ratchet; a new BT execution invariant test harness catches violations at CI time.
- **CaseLedger commit ordering and uniqueness** ([PR #1080](https://github.com/CERTCC/Vultron/pull/1080), [PR #962](https://github.com/CERTCC/Vultron/pull/962)): `commit_last` ordering fixed; CaseActor commit-path uniqueness ensures each logical assertion produces exactly one canonical `CaseLedgerEntry` regardless of fan-out.
- **Ledger-commit spec and CI** ([PR #1017](https://github.com/CERTCC/Vultron/pull/1017), [PR #936](https://github.com/CERTCC/Vultron/pull/936)): BT-06-006 (ledger-commit obligation on `SUCCESS`) encoded in `specs/behavior-tree-integration.yaml`; CI harness enforces the invariant for all BT node subclasses.
- **Routing-gated state mutation (BT-19)** ([PR #1203](https://github.com/CERTCC/Vultron/pull/1203)): New spec requirements BT-19-001/002 codify that lifecycle BT sequences must resolve routing prerequisites (Case Manager) in a read-only guard before applying state mutation, with a shared factory requirement. AGENTS.md Common Pitfall entry added.
