# Vultron Project — 2025 Annual Work Sponsor Report

## Executive Summary

2025 was a year of transformation for the Vultron project, spanning
infrastructure modernization in the first half and substantial new protocol
capability in the second half. The year's headline achievement was the
construction of a working REST API layer (v0.1 through v0.2.2) that implements
core CVD (Coordinated Vulnerability Disclosure) protocol message flows —
including report submission, offer negotiation, and actor inbox/outbox
management. Alongside this, the project modernized its entire development
environment: upgrading to Python 3.13, adopting `uv` as the package manager,
migrating to Pydantic v2, and establishing consistent code quality tooling.

---

## New Protocol API: From Stub to Working Message Flows

The most significant work of 2025 was the design and incremental delivery of a
FastAPI-based REST API implementing the Vultron protocol. Starting in September
with initial scaffolding, the team shipped four named releases between October
and November:

- **API v0.1.0** ([PR #251](https://github.com/CERTCC/Vultron/pull/251))
  established the API framework, actor registration, inbox handling, and a
  working Docker development environment.
- **API v0.2.0** ([PR #253](https://github.com/CERTCC/Vultron/pull/253))
  added a complete reporting workflow: handlers for the
  Create/Offer/Accept/Reject activity types, an in-memory data layer, and
  sequence-diagram documentation of the vulnerability reporting flow.
- **API v0.2.1** ([PR #262](https://github.com/CERTCC/Vultron/pull/262))
  introduced a cleaner internal data structure, a working "receive report"
  demo script, and additional unit test coverage for the backend.
- **API v0.2.2** ([PR #267](https://github.com/CERTCC/Vultron/pull/267))
  completed offer-lifecycle handling — including `Reject`,
  `TentativeReject`, and `Accept(Offer)` — and added actor inbox/outbox
  management endpoints.

By year-end, the API supports the full initial report intake lifecycle:
a finder can submit a vulnerability report; the receiving actor can accept,
tentatively reject, or reject it; and both sides maintain synchronized inbox
and outbox views of the exchange.

## Infrastructure Modernization

September 2025 saw a concentrated effort to modernize the project's runtime
and tooling foundations:

- **Python 3.13** is now the target version in CI
  ([PR #233](https://github.com/CERTCC/Vultron/pull/233)).
- **Pydantic v2** was fully adopted: data classes were refactored to use
  Pydantic models ([PR #240](https://github.com/CERTCC/Vultron/pull/240))
  and serialization aliases were updated to match the new API
  ([PR #242](https://github.com/CERTCC/Vultron/pull/242)).
- **`uv` package manager** replaced the legacy `requirements.txt`
  ([PR #241](https://github.com/CERTCC/Vultron/pull/241)), aligning
  dependency management with current Python ecosystem best practices.
- **Docker support** was introduced
  ([PR #242](https://github.com/CERTCC/Vultron/pull/242)), enabling
  `docker-compose up api-dev` as a one-command development environment, and
  subsequently expanded with live-reload mounts and a persistent virtual
  environment.

These changes collectively reduce setup friction for new contributors and
ensure compatibility with current Python and tooling releases.

## Code Quality and Developer Tooling

Alongside infrastructure upgrades, the project established a consistent code
quality baseline:

- **Black** was added as a pre-commit formatting hook, ensuring all Python
  source files are uniformly formatted before commit.
- **flake8** and **mypy** were integrated with Makefile targets, formalizing
  static analysis as part of the development workflow.
- A **Makefile** was introduced with standard targets (`lint`, `format`,
  `test`, API server startup), making common workflows self-documenting.
- **GitHub Copilot instructions** were added
  ([PR #248](https://github.com/CERTCC/Vultron/pull/248)), configuring the
  repository to guide AI-assisted development with project-specific coding
  conventions and architectural context.
- **Docstrings** across the codebase were reviewed and updated
  ([PR #232](https://github.com/CERTCC/Vultron/pull/232)), and enum utilities
  were adapted for Python 3.13 compatibility
  ([PR #231](https://github.com/CERTCC/Vultron/pull/231)).

## Documentation and Public Release

In the first quarter of 2025, the project published **Vultron v2024.4.3**
([PR #199](https://github.com/CERTCC/Vultron/pull/199)) — a documentation and
specification release making the latest protocol research publicly available.
Alongside this, the public documentation site received a visual refresh
([PR #198](https://github.com/CERTCC/Vultron/pull/198)), updating the template
and layout to improve readability. Inline documentation corrections (including
fixing the API name from "Vulcan" to "Vultron") were also applied during the
API development phase.

---

## Dependency Maintenance

Over 40 automated dependency updates were merged cleanly throughout 2025,
covering the `mkdocs` documentation toolchain, `pydantic`, `fastapi`,
`pandas`, `scipy`, `rdflib` (ontology), and GitHub Actions runners. This
ongoing hygiene work keeps the project's supply chain current and reduces
the accumulation of security and compatibility debt.
