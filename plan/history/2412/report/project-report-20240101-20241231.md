# Vultron 2024 Annual Project Report

**Period:** January 1, 2024 – December 31, 2024

## Executive Summary

In 2024, the Vultron project made substantial progress on two fronts: establishing
a formal protocol specification for ActivityStreams-based messaging, and significantly
expanding the project's publicly accessible knowledge base with content drawn from
published research on CVD state modeling. The year also saw the project adopt a
mature release-versioning scheme and invest in tooling that improves long-term
maintainability.

---

## Themes

### 1. ActivityStreams Protocol Foundation (January)

The most concentrated burst of work in 2024 came in early January with the release
of v2024.1. The team formally specified the message format that Vultron participants
will use to communicate — adopting the W3C ActivityStreams 2.0 standard as the wire
format for protocol messages. This effort produced:

- A formal Architecture Decision Record (ADR) documenting the choice of
  ActivityStreams vocabulary
- A Python implementation of the Vultron ActivityStreams vocabulary, including
  worked examples
- An OWL ontology capturing the formal vocabulary
- Narrative documentation describing the process flow

This work laid the foundation for all future protocol interoperability: any two
Vultron-compliant systems can now exchange structured messages using a defined,
well-understood format.

### 2. CVD Knowledge Base Expansion (April — v2024.4)

The largest single content addition of the year came in April with the v2024.4
release, which integrated material from an academic research paper on CVD state
modeling. This involved converting a technical report into structured documentation,
then editing, reorganizing, and cross-linking it across dozens of new pages. The
expanded knowledge base now covers:

- CVD stakeholder roles and their influence on outcomes
- Policy formalization for coordinated disclosure
- Situation awareness concepts
- Zero-day definitions and Vulnerability Equities Process (VEP) context
- Formal action rules and measurability of CVD practice

This expansion significantly improved the project's value as a public reference for
researchers and practitioners.

### 3. Visualization and Communication Aids (March)

To help readers understand how coordination and disclosure interact in the protocol,
the team added a set of coordination and disclosure diagrams. These visual aids
accompany the formal model and make the protocol more accessible to audiences without
deep technical background.

### 4. Versioning Governance — Adopting CalVer (April)

The team formally documented the switch to calendar-based versioning (CalVer),
captured in a new ADR. Releases now carry version strings in the form `YYYY.N`
(e.g., `v2024.4`), making the currency of a release immediately legible to external
consumers and sponsors. This was accompanied by minor documentation polish, TLP
classification notes, and navigation improvements.

### 5. Site Infrastructure and Automation (January–March)

Several quality-of-life improvements to the documentation site and CI/CD pipeline
were shipped across Q1:

- Added an automated link checker to catch broken references on every commit
- Fixed a site build bug that prevented the deployed site from reporting its own
  version correctly
- Repaired LaTeX math rendering in documentation pages
- Added site analytics
- Improved Dependabot configuration to group related dependency updates, reducing
  noise in the PR queue

### 6. Ongoing Dependency and Toolchain Maintenance (throughout)

The project's documentation toolchain (MkDocs, various plugins) and scientific
Python stack (scipy, pandas, networkx, rdflib) received continuous updates
throughout the year. These are routine but important: they keep the build
environment secure, reproducible, and current.
