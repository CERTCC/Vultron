Wire the call-out point abstraction (from #1151) into Vultron's demo BTs by: (1) auditing existing BT trees for baked-in policy nodes, (2) refactoring those nodes to explicit call-out points with deterministic backends, and (3) demonstrating the framework with one randomised (fuzzer-backend) demo run.

**Processed**: 2026-07-23 — Design gap identified during planning: the call-out point configuration system (how running code selects which backend factory to use at runtime) has not been designed. Design issue #1631 created to resolve the following questions before implementation:

- Selection mechanism (per-actor config, CLI args, registry)
- Granularity (modal / per-call-out-point / group-based / feature flags)
- Configuration surface (YAML / CLI / both)
- Extensibility model (how new backends register and validate contracts)
- Default direction (deterministic AlwaysSucceed/AlwaysFail vs probabilistic fuzzer)

Issue #1152 remains open, blocked by #1631. Once #1631 resolves the design, it will update #1152 with implementation details before #1152 closes.
