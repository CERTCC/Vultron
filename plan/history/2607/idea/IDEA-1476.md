## Summary

Generate rendered reference pages for `specs/rm-behavior.yaml`,
`specs/em-behavior.yaml`, and `specs/cs-behavior.yaml` under
`docs/reference/specs/` using the MkDocs `markdown-exec` plugin (already in
the plugin stack). Behavior logic docs can then link to these pages instead of
embedding static spec IDs as inline text.

## Motivation

Issue #1426 (annotate behavior logic docs with RMB/EMB/CSB spec IDs) was
originally planned to add static text citations to each behavior logic doc.
That creates a maintainability problem: any time a spec group ID is added,
renamed, or reordered, every doc that embeds that ID needs a manual update.
Linking to generated reference pages is far more durable — the page content
auto-updates from the YAML source.

The MkDocs stack already includes `markdown-exec`, which can run Python inline
in a `.md` page at build time. A page like `docs/reference/specs/rm-behavior.md`
can load `specs/rm-behavior.yaml` and render a structured table of all RMB
groups and items — no manual upkeep needed after initial setup.

## Rough Approach (from issue)

1. Prototype a `docs/reference/specs/rm-behavior.md` page that uses
   `markdown-exec` to load `specs/rm-behavior.yaml` and render a requirements
   table (group ID, title, trigger, items with priority + statement).
2. Confirm `mkdocs build --strict` passes with the page in the nav.
3. Repeat for `em-behavior.md` and `cs-behavior.md`.
4. Add the three pages to `mkdocs.yml` nav under `reference/`.
5. Update issue #1426 to reflect the new approach: behavior logic docs link to
   these generated pages (e.g., `../../../reference/specs/rm-behavior.md`)
   rather than embedding static IDs inline.

## Expanded Plan (from grill-me interview, 2026-07-17)

During planning, the design was broadened significantly from 3 pages to 6.
Key decisions:

- **6 kind pages** (not 65 per-file pages or 3 behavioral-only pages) —
  one page per `SpecKind` value: general, pattern, domain, language,
  implementation, dev-process. Kind is the discovery axis for docs readers.
- **markdown-exec** in each stub page calls a new `docs_render.py` helper.
  Drift is structurally impossible; rendered HTML IS the current YAML.
- **docs_render.py** (new, separate from render.py) — `render_for_kind()`
  produces rich MkDocs Material markdown: priority badges, group summary
  tables, collapsible behavioral spec details blocks, cross-link anchors.
- **H2 = file title, H3 = group, specs under H3** within each kind page.
- **Spec ID cross-links** resolve to anchor URLs on the appropriate kind page
  (same-page `#ID` or relative `../kind/#ID`).
- **BEHAVIORAL tag** added to SpecTag enum; RMB/EMB/CSB tagged for visual
  distinction in domain.md.
- **Kind coverage test** in test/metadata/specs/ — all SpecKind values must
  have a corresponding docs/reference/specs/<kind>.md file.
- **Follow-on issue #1480**: tag population for all 65 spec files.
