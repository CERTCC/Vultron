# Vultron Project — February 2026 Work Sponsor Report

## Executive Summary

February 2026 was the most productive month for protocol implementation to
date. The team completed all eight phases of Behavior Tree (BT) integration —
a framework that automates protocol decision-making — achieving full coverage
of the Coordinated Vulnerability Disclosure (CVD) workflow. Alongside this,
twelve runnable end-to-end demo scenarios were produced and consolidated into a
single, easy-to-use command-line tool. The month closed with multiple
dependency updates and a comprehensive documentation pass.

---

## Behavior Tree Automation: Full Protocol Coverage

The core technical achievement of the month was completing all eight phases
(BT-1 through BT-8) of Behavior Tree integration across the Vultron protocol.

Behavior Trees are a pattern borrowed from game AI and robotics that express
complex conditional logic as a structured, inspectable tree rather than ad-hoc
code. Each node in the tree either succeeds, fails, or is still running —
making protocol behavior auditable and testable at a granular level.

By the end of the month, every major CVD protocol action was automated via
behavior trees:

- **Report validation** (BT-1): incoming vulnerability reports are validated
  and accepted or rejected through a structured policy tree. Measured at well
  under the 1ms target at the 99th percentile.
- **Case engagement and deferral** (BT-2.1): a participant's prioritization
  decision — engage or defer a case — now goes through a behavior tree that
  evaluates case priority (with a stub for SSVC-based scoring).
- **Case creation and lifecycle** (BT-3): case creation checks for duplicates,
  validates the case object, persists it, and notifies relevant actors — all
  in a single tree.
- **Actor invitation** (BT-4): the full invite/accept/reject cycle for adding
  participants to a case.
- **Embargo management** (BT-5): all seven embargo handlers — propose, accept,
  reject, announce, add, remove, and invite to embargo — are now wired and
  tested.
- **Status updates and notes** (BT-6): handlers for case notes and
  case/participant status updates.
- **Actor suggestions and ownership transfer** (BT-7): a coordinator can
  recommend a new actor to a case or transfer case ownership; accept and reject
  paths are both handled.
- **Missing message types** (BT-8): the `UPDATE_CASE` message type was fully
  wired. Two other candidate message types (`REENGAGE_CASE` and
  `CHOOSE_PREFERRED_EMBARGO`) were evaluated and consciously deferred, with
  rationale documented.

The test suite grew from 456 tests at the start of the month to 554 tests by
the end.

## Live End-to-End Demo Scenarios

Each Behavior Tree phase was paired with a runnable demo script that exercises
a real multi-actor CVD workflow against the live API. By month end, twelve
scenario demos existed:

- Report receipt and acknowledgment
- Case initialization and management
- Actor invitation (accept and reject paths)
- Embargo establishment (propose-accept and propose-reject paths)
- Status updates
- Actor suggestion and ownership transfer
- Embargo management and participant management

These demos serve as both acceptance tests and living documentation of how the
protocol behaves in practice.

See [PR #292](https://github.com/CERTCC/Vultron/pull/292) (Report Demo) and
[PR #293](https://github.com/CERTCC/Vultron/pull/293) (BT Demo Integration)
for the initial demo work; the full DEMO-3 series
([PR #303](https://github.com/CERTCC/Vultron/pull/303)) completed the
remaining scenario coverage.

## Unified Demo CLI

All twelve demo scripts were consolidated into a single `vultron-demo`
command-line tool ([PR #305](https://github.com/CERTCC/Vultron/pull/305)).
Key improvements:

- A single entry point (`vultron-demo`) with sub-commands for each scenario
  and an `all` option to run every demo in sequence.
- Shared utilities (HTTP helpers, context managers, data layer client) were
  extracted to a common module, eliminating duplication across scripts.
- Demo teardown is now guaranteed — even if a demo crashes, the environment
  resets cleanly.
- The Docker setup was simplified from one container per demo to a single
  `demo` service.
- Test execution speed improved approximately ten-fold by eliminating
  artificial time delays in the test suite.
- Integration test infrastructure added (`make integration-test`) for
  end-to-end validation.

## API Reliability and Bug Fixes

Several small but important reliability fixes were merged in the first half of
the month:

- Lazy initialization of the handler map was restored, preventing startup
  errors when the server is imported without all handlers loaded
  ([PR #289](https://github.com/CERTCC/Vultron/pull/289)).
- Duplicate test function names were corrected, ensuring the full test suite
  runs without shadowed tests
  ([PR #288](https://github.com/CERTCC/Vultron/pull/288)).
- A stale import was removed from the message rehydration module
  ([PR #286](https://github.com/CERTCC/Vultron/pull/286)).
- Actor invitation pattern matching was corrected — real actors have type
  "Organization" or "Person", not the generic "Actor" type
  ([PR #303](https://github.com/CERTCC/Vultron/pull/303)).
- The pattern-matching engine was fixed to handle string URI references before
  attempting structured object matching
  ([PR #303](https://github.com/CERTCC/Vultron/pull/303)).
